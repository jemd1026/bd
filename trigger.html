Introducción a los Triggers en SQL Server

Guía didáctica para alumnos

1. ¿Qué es un Trigger?

Un trigger, también llamado disparador o desencadenador, es un tipo especial de procedimiento almacenado que se ejecuta automáticamente cuando ocurre una acción sobre una tabla o vista.

En otras palabras, un trigger permite que la base de datos reaccione automáticamente cuando se insertan, actualizan o eliminan datos.

Idea clave: Un trigger no se ejecuta manualmente. Se activa automáticamente cuando ocurre un evento.

2. ¿Para qué sirve un Trigger?

Los triggers se utilizan principalmente para:

  • Registrar cambios realizados en una tabla.// Queremos saber cada vez que el precio de un producto cambia, guardando el valor anterior y el nuevo valor.
  • Crear auditorías de datos.// Queremos registrar quién insertó un producto y cuándo, sin modificar la tabla Productos.
  • Controlar reglas de negocio.// Queremos asegurarnos de que ciertas condiciones se cumplan antes de permitir una operación.
  • Evitar eliminaciones o modificaciones indebidas.// Queremos asegurarnos de que ciertos registros no se eliminen o modifiquen bajo ciertas condiciones.
  • Mantener la integridad y coherencia de la información.// Queremos asegurarnos de que los datos permanezcan consistentes y válidos.

Por ejemplo, si un usuario modifica el precio de un producto, un trigger puede guardar automáticamente el precio anterior, el nuevo precio, la fecha y el usuario que realizó el cambio.

3. ¿Cuándo se ejecuta un Trigger?

Un trigger se ejecuta cuando ocurre una acción específica sobre una tabla. Las acciones más comunes son:

  • INSERT: cuando se agrega un nuevo registro.
  • UPDATE: cuando se modifica un registro existente.
  • DELETE: cuando se elimina un registro.

Ejemplo sencillo:

-- Este INSERT dispara cualquier trigger AFTER INSERT sobre Productos
INSERT INTO Productos (nombre, precio)
VALUES ('Mouse inalámbrico', 35.00);

Si la tabla Productos tiene un trigger para INSERT, este se ejecutará automáticamente después de insertar el producto.

4. Diferencia entre Trigger, Stored Procedure, Constraint y Default

Elemento ¿Qué es? ¿Cuándo se ejecuta? Ejemplo de uso
Trigger Código que se ejecuta automáticamente ante un evento. Cuando ocurre un INSERT, UPDATE o DELETE. Guardar historial cuando cambia el precio de un producto.
Stored Procedure Conjunto de instrucciones SQL guardadas en la base de datos. Cuando el usuario o sistema lo llama manualmente. Registrar una venta mediante un procedimiento.
Constraint Regla que limita o valida los datos de una tabla. Cuando se insertan o modifican datos. Evitar que el precio sea menor que cero.// CREATE TABLE Productos (..., Precio DECIMAL(10,2) CHECK (Precio >= 0), ...);
Default Valor automático asignado a una columna si no se indica uno. Cuando se inserta un nuevo registro. Asignar la fecha actual a una venta. // CREATE TABLE Productos (..., FechaRegistro DATETIME DEFAULT GETDATE(), ...);

5. Tipos de Triggers

En SQL Server existen dos grandes tipos de triggers según cuándo se ejecutan en relación con la operación que los dispara:

AFTER Trigger — se ejecuta después de que la operación se completa.

La operación (INSERT, UPDATE o DELETE) ocurre primero; una vez que los datos ya fueron modificados en la tabla, el trigger entra en acción.

  • AFTER INSERT // Se activa luego de insertar un registro → útil para auditoría.
  • AFTER UPDATE // Se activa luego de modificar un registro → útil para guardar historial de cambios.
  • AFTER DELETE // Se activa luego de eliminar un registro → útil para limpiar datos relacionados.
Analogía: Es como la alarma de una caja fuerte que suena después de que alguien ya la abrió. El evento ya ocurrió; el trigger reacciona ante eso.

INSTEAD OF Trigger — se ejecuta en lugar de la operación original.

La operación solicitada (INSERT, UPDATE o DELETE) no llega a ejecutarse. El trigger la intercepta y decide qué hacer.

  • INSTEAD OF DELETE // Alguien intenta borrar un cliente activo → el trigger cancela la eliminación y muestra un mensaje.
  • INSTEAD OF INSERT // Alguien intenta insertar datos en una vista → el trigger redirige la inserción a las tablas reales.
Analogía: Es como un guardia de seguridad que intercepta a alguien antes de que entre. La acción nunca ocurrió; el trigger decide si se permite o no.
Tipo ¿Cuándo actúa? ¿La operación ocurre? Uso típico
AFTER Después del INSERT / UPDATE / DELETE Sí, ya ocurrió Auditoría, historial, limpieza de datos relacionados
INSTEAD OF En lugar del INSERT / UPDATE / DELETE No — el trigger la reemplaza Bloquear eliminaciones, controlar inserciones en vistas

6. Tablas especiales: inserted y deleted

Dentro de cualquier trigger, SQL Server crea automáticamente dos tablas temporales que solo existen mientras el trigger se está ejecutando. Son la clave para entender cómo acceder a los datos que cambiaron.

Regla de oro: Estas tablas tienen exactamente las mismas columnas que la tabla original. No hay que crearlas — SQL Server las genera solo.

6.1 Tabla inserted — los valores nuevos

Contiene los registros tal como quedarán después de la operación. Existe en triggers de INSERT y UPDATE.

Operación que la genera:
INSERT INTO Clientes (nombre) VALUES ('Juan');

Qué contiene inserted dentro del trigger:

idnombre
5Juan

→ El trigger puede leer inserted.nombre para saber qué se acaba de agregar.

6.2 Tabla deleted — los valores anteriores

Contiene los registros tal como estaban antes de la operación. Existe en triggers de DELETE y UPDATE.

Operación que la genera:
DELETE FROM Clientes WHERE id = 1;

Qué contiene deleted dentro del trigger:

idnombre
1Juan

→ El trigger puede leer deleted.nombre para saber qué se acaba de eliminar.

6.3 En un UPDATE — se usan las dos al mismo tiempo

Este es el caso más importante y el que más confunde al principio. Un UPDATE es internamente un DELETE del valor viejo + un INSERT del valor nuevo, por eso SQL Server llena ambas tablas:

Operación:
UPDATE Productos SET stock = 5 WHERE id = 1; -- El stock antes era 10

deleted — valor anterior

idstock
110
→

inserted — valor nuevo

idstock
15
¿Cómo lo usa el trigger?
// Para calcular cuánto bajó el stock:
SELECT d.id,
       d.stock AS stock_anterior,
       i.stock AS stock_nuevo,
       d.stock - i.stock AS diferencia
FROM deleted d
JOIN inserted i ON d.id = i.id;

6.4 Resumen: ¿qué tabla existe en cada operación?

Operación inserted deleted ¿Qué contiene cada una?
INSERT ✔ Sí — No inserted: los registros recién agregados
DELETE — No ✔ Sí deleted: los registros que se eliminaron
UPDATE ✔ Sí ✔ Sí deleted: valores anteriores  |  inserted: valores nuevos
Para la clase: conviene dibujar en la pizarra las dos tablas lado a lado cuando se explica el UPDATE. Es el concepto que más cuesta visualizar y el que más se usa en triggers reales de auditoría e historial.

7. Sintaxis básica de un Trigger

Todo trigger en SQL Server sigue la misma estructura. Analicémosla parte por parte:

CREATE TRIGGER nombre_trigger
ON tabla
AFTER INSERT
AS
BEGIN
    -- instrucciones que se ejecutan automáticamente
END;
Palabra clave ¿Qué indica? Ejemplo concreto
CREATE TRIGGER Crea un nuevo trigger y le da un nombre. CREATE TRIGGER trg_nuevo_producto
ON Indica en qué tabla o vista estará vigilando. ON Productos
AFTER / INSTEAD OF Define el tipo de trigger y el evento que lo dispara. AFTER INSERT  |  INSTEAD OF DELETE
AS Marca el inicio del cuerpo del trigger (lo que hará). Siempre va después de AFTER/INSTEAD OF
BEGIN … END Delimitadores del bloque de instrucciones. Todo lo que va entre ellos se ejecuta. Como las llaves { } en otros lenguajes
Convención de nombres: En SQL Server es buena práctica nombrar los triggers con el prefijo trg_ seguido de la tabla y la acción, por ejemplo trg_productos_insert. Esto facilita identificarlos en la base de datos.

8. Primer ejemplo de Trigger

Antes de escribir cualquier trigger necesitamos una base de datos de trabajo. El siguiente script crea todo desde cero: la base de datos, las cuatro tablas y algunos datos iniciales para probar.

Contexto: A lo largo de los ejemplos nos centraremos principalmente en Productos y DetalleVentas, que son las tablas donde ocurren los eventos más interesantes para un trigger.

Paso 1 — Crear la base de datos

-- Si ya existe, la eliminamos y empezamos limpio
IF EXISTS (SELECT name FROM sys.databases WHERE name = 'Tienda')
    DROP DATABASE Tienda;

CREATE DATABASE Tienda;
GO

USE Tienda;
GO

Paso 2 — Crear las cuatro tablas

Las relaciones entre tablas se declaran con REFERENCES (forma corta de FOREIGN KEY). Garantizan que no pueda existir una venta sin cliente, ni un detalle sin producto.

-- ① Productos — no depende de ninguna otra tabla (tabla padre)
CREATE TABLE Productos (
    id       INT IDENTITY(1,1) PRIMARY KEY, -- PK: id autonumérico
    nombre   VARCHAR(100) NOT NULL,
    precio   DECIMAL(10,2) NOT NULL,
    stock    INT NOT NULL DEFAULT 0,   -- el trigger lo descontará al vender
    activo   BIT NOT NULL DEFAULT 1   -- 1=activo, 0=inactivo
);
GO

-- ② Clientes — no depende de ninguna otra tabla (tabla padre)
CREATE TABLE Clientes (
    id       INT IDENTITY(1,1) PRIMARY KEY, -- PK: id autonumérico
    nombre   VARCHAR(100) NOT NULL,
    email    VARCHAR(150)
);
GO

-- ③ Ventas — depende de Clientes (tabla hija)
CREATE TABLE Ventas (
    id           INT IDENTITY(1,1) PRIMARY KEY,
    id_cliente   INT NOT NULL REFERENCES Clientes(id), -- FK → Clientes: no puede existir venta sin cliente
    fecha        DATETIME NOT NULL DEFAULT GETDATE(), -- fecha automática al insertar
    total        DECIMAL(12,2) NOT NULL DEFAULT 0 -- un trigger podría actualizarlo
);
GO

-- ④ DetalleVentas — depende de Ventas y de Productos (tabla hija de ambas)
CREATE TABLE DetalleVentas (
    id           INT IDENTITY(1,1) PRIMARY KEY,
    id_venta     INT NOT NULL REFERENCES Ventas(id),    -- FK → Ventas: cada línea pertenece a una venta
    id_producto  INT NOT NULL REFERENCES Productos(id), -- FK → Productos: no puede venderse un producto inexistente
    cantidad     INT NOT NULL,                        -- el trigger usará esto para descontar stock
    precio_unit  DECIMAL(10,2) NOT NULL                 -- precio al momento de la venta (puede cambiar después)
);
GO
Relación entre las tablas:
Clientes (id) ←── Ventas (id_cliente)
Ventas (id)    ←── DetalleVentas (id_venta)
Productos (id) ←── DetalleVentas (id_producto)

Los triggers que veremos actúan principalmente sobre Productos (cambios de precio, stock, protección) y DetalleVentas (descuento automático de stock al vender).

Paso 3 — Insertar datos de prueba

-- Productos
INSERT INTO Productos (nombre, precio, stock) VALUES
    ('Teclado USB',    25.00, 50),
    ('Mouse inalámbrico', 35.00, 30),
    ('Monitor 24"',   320.00, 10),

-- Clientes
INSERT INTO Clientes (nombre, email) VALUES
    ('Ana Torres',  'ana@mail.com'),
    ('Luis Ramos',  'luis@mail.com'),
    ('María López', 'maria@mail.com');

-- Una venta de prueba
INSERT INTO Ventas (id_cliente) VALUES (1);

-- Detalle: Ana compró 2 teclados y 1 mouse
INSERT INTO DetalleVentas (id_venta, id_producto, cantidad, precio_unit) VALUES
    (1, 1, 2, 25.00),
    (1, 2, 1, 35.00);
GO

Paso 4 — El primer trigger sobre esta base de datos

Con la base de datos lista, el primer trigger simplemente confirma que un producto fue insertado. Es el "Hola mundo" de los triggers.

CREATE TRIGGER trg_producto_insertado
ON Productos
AFTER INSERT
AS
BEGIN
    PRINT '✔ Se ha insertado un nuevo producto en el catálogo.';
END;
GO

-- Prueba: este INSERT dispara el trigger automáticamente
INSERT INTO Productos (nombre, precio, stock)
VALUES ('Webcam HD', 75.00, 15);
Para recordar: Los triggers son útiles, pero deben usarse con cuidado porque se ejecutan automáticamente y pueden afectar el rendimiento si contienen mucha lógica. En los ejemplos de la sección 9 veremos casos reales sobre estas mismas tablas.

9. Ejemplos prácticos: un trigger para cada propósito

Un ejemplo práctico para cada uno de los cinco usos principales de los triggers. Todos trabajan sobre la base de datos Tienda creada en la sección 8.

9.1 Registrar cambios realizados en una tabla

Situación: Queremos saber cada vez que el precio de un producto cambia, guardando el valor anterior y el nuevo valor.

-- Tabla auxiliar: guarda el historial de cambios de precio
CREATE TABLE HistorialPrecios (
    id_producto  INT,
    precio_antes DECIMAL(10,2),
    precio_nuevo DECIMAL(10,2),
    fecha        DATETIME DEFAULT GETDATE() -- se llena automáticamente
);
GO

-- Trigger: se dispara cada vez que se modifica un precio en Productos
CREATE TRIGGER trg_registrar_cambio_precio
ON Productos
AFTER UPDATE -- actúa después del UPDATE, cuando los datos ya cambiaron
AS
BEGIN
    INSERT INTO HistorialPrecios (id_producto, precio_antes, precio_nuevo)
    SELECT d.id, d.precio, i.precio -- deleted = antes, inserted = después
    FROM deleted d
    JOIN inserted i ON d.id = i.id
    WHERE d.precio <> i.precio; -- solo guarda si el precio realmente cambió
END;
GO

-- Prueba: cambia el precio del Teclado USB de 25.00 a 29.00
UPDATE Productos SET precio = 29.00 WHERE id = 1;
Resultado: Si ejecutamos UPDATE Productos SET precio = 45.00 WHERE id = 1;, el trigger guarda automáticamente el precio anterior y el nuevo en HistorialPrecios.

9.2 Crear auditorías de datos

Situación: Queremos registrar quién insertó un producto y cuándo, sin modificar la tabla Productos.

-- Tabla auxiliar: registra quién hizo qué y cuándo
CREATE TABLE AuditoriaProductos (
    accion      VARCHAR(10),                       -- 'INSERT', 'UPDATE' o 'DELETE'
    id_producto INT,
    nombre      VARCHAR(100),
    usuario     VARCHAR(100) DEFAULT SYSTEM_USER, -- usuario de SQL Server conectado
    fecha       DATETIME    DEFAULT GETDATE()  -- fecha y hora exacta
);
GO

-- Trigger: cada vez que se agrega un producto, queda registrado en auditoría
CREATE TRIGGER trg_auditoria_insercion
ON Productos
AFTER INSERT
AS
BEGIN
    INSERT INTO AuditoriaProductos (accion, id_producto, nombre)
    SELECT 'INSERT', id, nombre -- inserted tiene los datos recién agregados
    FROM inserted;
END;
GO

-- Prueba: insertar un producto nuevo dispara el trigger
INSERT INTO Productos (nombre, precio, stock)
VALUES ('Webcam HD', 75.00, 15);

-- Verificar qué quedó registrado en auditoría
SELECT * FROM AuditoriaProductos;
Resultado: Cada producto nuevo queda registrado en AuditoriaProductos con el nombre del usuario de base de datos y la fecha exacta de la inserción. SYSTEM_USER devuelve el usuario conectado automáticamente.

9.3 Controlar reglas de negocio

Situación: La empresa decide que ningún producto puede tener un precio mayor a 10 000. Si alguien intenta asignarlo, el trigger debe cancelar la operación.

-- Trigger: cancela el UPDATE si el precio nuevo supera 10 000
CREATE TRIGGER trg_precio_maximo
ON Productos
AFTER UPDATE
AS
BEGIN
    IF EXISTS (SELECT 1 FROM inserted WHERE precio > 10000) -- inserted tiene el precio nuevo
    BEGIN
        ROLLBACK TRANSACTION; -- deshace el UPDATE completo
        RAISERROR('El precio no puede superar 10 000.', 16, 1); -- muestra el error
    END
END;
GO

-- Prueba 1: precio válido → se permite
UPDATE Productos SET precio = 350.00 WHERE id = 3;

-- Prueba 2: precio inválido → el trigger lo cancela con error
UPDATE Productos SET precio = 99999.00 WHERE id = 3;
Resultado: Si alguien ejecuta UPDATE Productos SET precio = 99999 WHERE id = 2;, la transacción se cancela con ROLLBACK y aparece el mensaje de error. El precio no cambia en la tabla.

9.4 Evitar eliminaciones o modificaciones indebidas

Situación: Los productos marcados como activos (activo = 1) no deben poder eliminarse mientras sigan en uso.

-- Trigger: intercepta el DELETE y decide si se permite o no
CREATE TRIGGER trg_proteger_eliminacion
ON Productos
INSTEAD OF DELETE -- reemplaza al DELETE original; sin este trigger el DELETE ocurriría directamente
AS
BEGIN
    IF EXISTS (SELECT 1 FROM deleted WHERE activo = 1) -- deleted tiene el registro que querían borrar
    BEGIN
        RAISERROR('No se puede eliminar un producto activo.', 16, 1);
    END
    ELSE
    BEGIN
        -- El producto está inactivo: se permite la eliminación
        DELETE FROM Productos
        WHERE id IN (SELECT id FROM deleted);
    END
END;
GO

-- Prueba 1: intentar borrar un producto activo → el trigger lo bloquea
DELETE FROM Productos WHERE id = 1; -- activo = 1, se esperan error

-- Prueba 2: desactivar primero, luego borrar → el trigger lo permite
UPDATE Productos SET activo = 0 WHERE id = 1;
DELETE FROM Productos WHERE id = 1; -- ahora sí procede
Nota clave: Este trigger usa INSTEAD OF DELETE, lo que significa que reemplaza la operación DELETE en lugar de ejecutarse después. Así tiene control total sobre si se permite o no la eliminación.

9.5 Mantener la integridad y coherencia de la información

Situación: Cuando un producto se elimina, sus registros en HistorialPrecios también deben eliminarse para no dejar datos huérfanos.

-- Trigger: cuando se elimina un producto, limpia su historial automáticamente
CREATE TRIGGER trg_limpiar_historial
ON Productos
AFTER DELETE -- se ejecuta después de que el producto ya fue eliminado
AS
BEGIN
    DELETE FROM HistorialPrecios
    WHERE id_producto IN (SELECT id FROM deleted); -- deleted tiene el id del producto eliminado
END;
GO

-- Prueba: eliminar el Mouse inalámbrico (id=2) borra también su historial de precios
UPDATE Productos SET activo = 0 WHERE id = 2; -- primero lo desactivamos
DELETE FROM Productos WHERE id = 2;               -- esto dispara el trigger

-- Verificar que no quedaron registros huérfanos
SELECT * FROM HistorialPrecios WHERE id_producto = 2; -- debe devolver 0 filas
Resultado: Al eliminar el producto con id = 3, el trigger borra automáticamente todos los registros de precio que le pertenecían en HistorialPrecios, evitando que queden datos sin dueño (huérfanos).

Tablas especiales inserted y deleted:
Dentro de cualquier trigger, SQL Server crea automáticamente dos tablas temporales:
• inserted: contiene los registros nuevos (INSERT) o los valores nuevos (UPDATE).
• deleted: contiene los registros eliminados (DELETE) o los valores anteriores (UPDATE).
