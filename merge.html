La sentencia MERGE en SQL Server

Guía didáctica para alumnos

1. ¿Qué es MERGE?

MERGE es una sentencia SQL que permite realizar tres operaciones en una sola instrucción: insertar registros nuevos, actualizar los existentes y eliminar los que ya no corresponden.

La idea central es comparar una tabla origen (la fuente de datos nueva) contra una tabla destino (la que queremos mantener actualizada), y aplicar automáticamente lo que corresponda según el resultado de esa comparación.

Analogía: Imagina que tienes una lista de empleados en papel (origen) y una lista en el sistema (destino). MERGE revisa ambas y decide: si el empleado ya existe → actualiza sus datos; si es nuevo → lo agrega; si ya no aparece en el papel → lo elimina del sistema.

2. ¿Para qué sirve MERGE?

MERGE se utiliza principalmente para:

  • Sincronizar dos tablas que deben mantenerse iguales. // Ejemplo: sincronizar inventario de una sucursal con la central.
  • Cargar datos desde un sistema externo (ETL / importaciones). // Ejemplo: importar clientes desde un archivo Excel cada semana.
  • Reemplazar combinaciones de INSERT + UPDATE + DELETE en un solo paso. // Evita escribir tres sentencias separadas con lógica condicional.
  • Actualizar un resumen o tabla de totales cuando cambia el detalle. // Ejemplo: recalcular el stock al procesar una nueva lista de productos.

3. Ventajas y desventajas

Ventajas
  • Eficiencia: realiza INSERT + UPDATE + DELETE en una sola pasada sobre los datos, reduciendo el número de lecturas a la base de datos.
  • Atomicidad: toda la operación ocurre como una unidad — o todo se aplica o nada. Garantiza consistencia aunque ocurra un error a mitad del proceso.
  • Legibilidad: expresa en un solo bloque lógico lo que de otra forma requeriría tres sentencias separadas con condiciones IF EXISTS.
  • Menos bloqueos: al trabajar en una sola sentencia, el tiempo de bloqueo sobre las tablas involucradas tiende a ser menor que con múltiples operaciones individuales.
Desventajas
  • Complejidad: la sintaxis es más extensa y puede resultar difícil de leer para alumnos o desarrolladores que recién comienzan.
  • Problemas de rendimiento en tablas grandes: en conjuntos de millones de filas, MERGE puede ser más lento que operaciones individuales optimizadas con índices específicos.
  • Concurrencia: en entornos con muchas escrituras simultáneas, puede generar bloqueos muertos (deadlocks) si no se configura correctamente.
  • Compatibilidad: la sintaxis varía entre motores (SQL Server, MySQL, Oracle). Lo que funciona aquí puede no funcionar igual en otro sistema.

4. Sintaxis básica de MERGE

La estructura siempre sigue el mismo patrón de cinco partes:

MERGE INTO TablaDestino AS destino            -- tabla que queremos actualizar
USING TablaOrigen AS origen                -- tabla con los datos nuevos
ON destino.id = origen.id                   -- condición de coincidencia (clave de comparación)

WHEN MATCHED THEN                          -- el registro existe en ambas tablas → actualizar
    UPDATE SET destino.col = origen.col

WHEN NOT MATCHED BY TARGET THEN           -- existe en origen pero NO en destino → insertar
    INSERT (col1, col2) VALUES (origen.col1, origen.col2)

WHEN NOT MATCHED BY SOURCE THEN          -- existe en destino pero NO en origen → eliminar
    DELETE;
GO
Cláusula ¿Qué pregunta? ¿Qué hace?
MERGE INTO … AS destino ¿Cuál es la tabla que quiero mantener actualizada? Declara la tabla destino y le da un alias corto.
USING … AS origen ¿De dónde vienen los datos nuevos? Declara la fuente — puede ser una tabla, vista o subconsulta.
ON ¿Cómo sé si un registro de origen ya existe en destino? Define la condición de comparación, normalmente por clave primaria.
WHEN MATCHED ¿El registro existe en ambas tablas? Ejecuta un UPDATE para actualizar los campos que cambiaron.
WHEN NOT MATCHED BY TARGET ¿El registro está en origen pero falta en destino? Ejecuta un INSERT para agregarlo.
WHEN NOT MATCHED BY SOURCE ¿El registro está en destino pero ya no está en origen? Ejecuta un DELETE para eliminarlo.
Importante: Las tres cláusulas WHEN son opcionales. Puedes usar solo las que necesites. Por ejemplo, si solo quieres insertar y actualizar pero nunca eliminar, simplemente omites el WHEN NOT MATCHED BY SOURCE.

5. Las tres situaciones posibles — visualizadas

La clave para entender MERGE es visualizar qué puede pasar al comparar cada fila de la tabla origen con la tabla destino:

MATCHED

El registro existe en ambas tablas.
El id aparece en origen Y en destino.

→ Se ejecuta UPDATE // Los datos del origen reemplazan a los del destino.
NOT MATCHED BY TARGET

El registro es nuevo — solo existe en origen.
El id NO aparece en destino todavía.

→ Se ejecuta INSERT // Se agrega el registro a la tabla destino.
NOT MATCHED BY SOURCE

El registro ya no existe en origen.
El id está en destino pero desapareció del origen.

→ Se ejecuta DELETE // Se elimina de la tabla destino.

7. El MERGE completo — las tres cláusulas

Con los datos cargados, ejecutamos MERGE para que Inventario_Central quede sincronizado con Inventario_Nuevo:

-- MERGE completo: actualiza, inserta y elimina en un solo paso
MERGE INTO Inventario_Central AS destino             -- tabla que queremos actualizar
USING Inventario_Nuevo AS origen                   -- tabla con los datos nuevos
ON destino.id = origen.id                          -- comparamos por id (clave primaria)

-- ① El id existe en ambas tablas → actualizar precio y stock
WHEN MATCHED THEN
    UPDATE SET
        destino.nombre = origen.nombre,
        destino.precio = origen.precio,
        destino.stock = origen.stock

-- ② El id está en origen pero no en destino → insertar como nuevo
WHEN NOT MATCHED BY TARGET THEN
    INSERT (id, nombre, precio, stock)
    VALUES (origen.id, origen.nombre, origen.precio, origen.stock)

-- ③ El id está en destino pero ya no aparece en origen → eliminar
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;
GO

-- Verificar el resultado final
SELECT * FROM Inventario_Central ORDER BY id; -- debe tener ids 1-6, 9-13 (7 y 8 eliminados)
Resultado esperado: Después del MERGE, Inventario_Central debe tener 11 filas (ids 1–6, 9–13). Los ids 7 y 8 desaparecieron, los ids 11, 12 y 13 se agregaron, y los precios/stocks de 1, 2 y 9 se actualizaron.

8. MERGE parcial — solo algunas cláusulas

No siempre se necesitan las tres cláusulas. MERGE es flexible: puedes usar solo las que la situación requiera.

8.1 Solo INSERT y UPDATE (sin eliminar)

Situación: Llega una lista de productos nuevos y actualizados, pero no queremos eliminar ninguno del sistema aunque ya no aparezca en la lista.

-- MERGE sin DELETE: agrega nuevos y actualiza existentes, pero nunca borra
MERGE INTO Inventario_Central AS destino
USING Inventario_Nuevo AS origen
ON destino.id = origen.id

WHEN MATCHED THEN                                          -- actualiza si ya existe
    UPDATE SET destino.precio = origen.precio, destino.stock = origen.stock

WHEN NOT MATCHED BY TARGET THEN                              -- inserta si es nuevo
    INSERT (id, nombre, precio, stock) VALUES (origen.id, origen.nombre, origen.precio, origen.stock);
-- sin WHEN NOT MATCHED BY SOURCE → los registros que no están en origen se conservan
GO

8.2 UPDATE solo cuando algo realmente cambió

Situación: Queremos evitar actualizar registros que no cambiaron, para no disparar triggers innecesarios ni generar entradas falsas en el historial de cambios.

-- WHEN MATCHED AND ... → agrega una condición extra antes de actualizar
MERGE INTO Inventario_Central AS destino
USING Inventario_Nuevo AS origen
ON destino.id = origen.id

WHEN MATCHED AND -- solo actúa si precio O stock realmente cambiaron
    (destino.precio <> origen.precio OR destino.stock <> origen.stock) THEN
    UPDATE SET destino.precio = origen.precio, destino.stock = origen.stock

WHEN NOT MATCHED BY TARGET THEN
    INSERT (id, nombre, precio, stock) VALUES (origen.id, origen.nombre, origen.precio, origen.stock);
GO
¿Por qué es útil el AND? Sin esa condición, MERGE ejecuta el UPDATE aunque los valores sean idénticos, lo que genera escrituras innecesarias y puede disparar triggers de auditoría para filas que no cambiaron.

9. Capturar qué hizo MERGE con OUTPUT

MERGE puede reportar exactamente qué acción tomó con cada fila usando la cláusula OUTPUT. Es muy útil para auditoría o para verificar resultados en clase.

-- OUTPUT muestra qué acción tomó MERGE con cada fila
MERGE INTO Inventario_Central AS destino
USING Inventario_Nuevo AS origen
ON destino.id = origen.id

WHEN MATCHED AND (destino.precio <> origen.precio OR destino.stock <> origen.stock) THEN
    UPDATE SET destino.precio = origen.precio, destino.stock = origen.stock

WHEN NOT MATCHED BY TARGET THEN
    INSERT (id, nombre, precio, stock) VALUES (origen.id, origen.nombre, origen.precio, origen.stock)

WHEN NOT MATCHED BY SOURCE THEN
    DELETE

-- OUTPUT: $action dice si fue INSERT, UPDATE o DELETE
OUTPUT
    $action AS accion,                           -- 'INSERT', 'UPDATE' o 'DELETE'
    COALESCE(deleted.id, inserted.id) AS id,
    deleted.precio AS precio_antes,              -- NULL si fue INSERT
    inserted.precio AS precio_despues           -- NULL si fue DELETE
ORDER BY accion, id;
GO
Resultado esperado de OUTPUT:
Las filas aparecerán agrupadas por acción. Las eliminadas mostrarán precio_antes y precio_despues = NULL. Las insertadas mostrarán precio_antes = NULL y el nuevo precio. Las actualizadas mostrarán ambos precios para comparar.

10. MERGE vs INSERT + UPDATE + DELETE por separado

¿Cuándo conviene usar MERGE y cuándo es mejor hacerlo por separado?

Criterio MERGE Sentencias separadas
Código más corto ✔ Sí Requiere tres bloques con condiciones IF EXISTS
Atomicidad ✔ Una sola transacción Hay que envolverlas en BEGIN TRANSACTION manualmente
Rendimiento en tablas pequeñas ✔ Mejor o igual Similar
Rendimiento en tablas muy grandes Puede ser más lento sin índices adecuados ✔ Más controlable
Facilidad de depuración Más difícil de aislar errores ✔ Más fácil: se prueba cada operación
Uso con OUTPUT / auditoría ✔ Nativo con OUTPUT $action Requiere lógica adicional en cada sentencia
Recomendación práctica: Usa MERGE cuando necesitas sincronizar dos tablas o cargar datos externos y quieres hacerlo limpio en un solo bloque. Usa sentencias separadas cuando el volumen de datos es muy grande o cuando cada operación necesita lógica muy específica y diferente.

11. Resumen visual

Situación al comparar Cláusula WHEN Acción ¿Qué tablas existen dentro?
id en origen Y en destino WHEN MATCHED UPDATE inserted (nuevo) y deleted (anterior)
id en origen, NO en destino WHEN NOT MATCHED BY TARGET INSERT Solo inserted
id en destino, NO en origen WHEN NOT MATCHED BY SOURCE DELETE Solo deleted
Conexión con Triggers: Cuando MERGE actúa sobre una tabla que tiene triggers, cada acción individual (INSERT, UPDATE, DELETE) dispara su trigger correspondiente. Las tablas inserted y deleted dentro de esos triggers funcionan exactamente igual que con sentencias separadas.

Ejercicio 1 — Sincronizar inventario sin eliminar descontinuados

Enunciado:

Usando las tablas Inventario_Central e Inventario_Nuevo de la base de datos TiendaMERGE, escribir un MERGE que:

  • Actualice precio y stock de los productos que ya existen.
  • Inserte los productos nuevos.
  • No elimine ningún producto aunque no aparezca en la lista nueva.
  • Solo actualice si el precio o el stock realmente cambiaron.
Base de datos: TiendaMERGE
Pistas:
  • Omite la cláusula WHEN NOT MATCHED BY SOURCE para no eliminar.
  • Usa WHEN MATCHED AND (destino.precio <> origen.precio OR destino.stock <> origen.stock) para filtrar.
  • Agrega OUTPUT $action para ver qué hizo MERGE con cada fila.
Ver solución
USE TiendaMERGE;
GO

-- MERGE sin DELETE y con filtro de cambios reales
MERGE INTO Inventario_Central AS destino
USING Inventario_Nuevo AS origen
ON destino.id = origen.id

-- Solo actualiza si algo realmente cambió
WHEN MATCHED AND
    (destino.precio <> origen.precio OR destino.stock <> origen.stock) THEN
    UPDATE SET
        destino.nombre = origen.nombre,
        destino.precio = origen.precio,
        destino.stock  = origen.stock

-- Producto nuevo → insertar
WHEN NOT MATCHED BY TARGET THEN
    INSERT (id, nombre, precio, stock)
    VALUES (origen.id, origen.nombre, origen.precio, origen.stock)

-- Sin WHEN NOT MATCHED BY SOURCE → no se elimina nada

OUTPUT
    $action AS accion,
    COALESCE(inserted.id, deleted.id) AS id,
    deleted.precio AS precio_antes,
    inserted.precio AS precio_despues;
GO

-- Verificar: los ids 7 y 8 deben seguir existiendo (no se eliminaron)
SELECT * FROM Inventario_Central ORDER BY id;

Ejercicio 2 — Desactivar en vez de eliminar

Enunciado:

Modificar el MERGE completo (sección 7) para que en vez de eliminar los productos que ya no están en la lista nueva, los marque como activo = 0 (inactivos).

  • Los productos que coinciden → actualizar precio y stock.
  • Los productos nuevos → insertar con activo = 1.
  • Los productos que ya no están en origen → en vez de DELETE, hacer UPDATE SET activo = 0.
Base de datos: TiendaMERGE
Pistas:
  • En WHEN NOT MATCHED BY SOURCE puedes usar UPDATE en vez de DELETE.
  • Agrega AND destino.activo = 1 para no volver a "desactivar" uno que ya lo está.
  • Usa OUTPUT con $action para verificar que diga UPDATE en lugar de DELETE.
Ver solución
USE TiendaMERGE;
GO

-- Primero recargar datos limpios (ejecutar sección 6 de nuevo si es necesario)

MERGE INTO Inventario_Central AS destino
USING Inventario_Nuevo AS origen
ON destino.id = origen.id

-- Coinciden → actualizar y asegurar que quede activo
WHEN MATCHED THEN
    UPDATE SET
        destino.nombre = origen.nombre,
        destino.precio = origen.precio,
        destino.stock = origen.stock,
        destino.activo = 1 -- reactivar si estaba inactivo

-- Nuevo → insertar como activo
WHEN NOT MATCHED BY TARGET THEN
    INSERT (id, nombre, precio, stock, activo)
    VALUES (origen.id, origen.nombre, origen.precio, origen.stock, 1)

-- Ya no está en origen → desactivar en vez de eliminar
WHEN NOT MATCHED BY SOURCE AND destino.activo = 1 THEN -- solo si aún está activo
    UPDATE SET destino.activo = 0 -- marcar como inactivo (no eliminar)

OUTPUT
    $action AS accion,
    COALESCE(inserted.id, deleted.id) AS id,
    inserted.activo AS activo_despues;
GO

-- Verificar: ids 7 y 8 deben tener activo = 0 (no fueron borrados)
SELECT id, nombre, activo FROM Inventario_Central ORDER BY id;

Ejercicio 3 — Sincronizar lista de precios de productos (Northwind)

Enunciado:

El departamento de compras envía una lista actualizada de precios para algunos productos. Crea una tabla temporal #ListaPrecios con los nuevos precios y usa MERGE para actualizar Products:

  • Si el producto existe en ambas tablas → actualizar UnitPrice solo si el precio cambió.
  • Si el producto existe en #ListaPrecios pero no en Products → insertar como producto nuevo.
  • No eliminar ningún producto que no esté en la lista.
  • Usar OUTPUT para mostrar qué productos se actualizaron y cuáles se insertaron.
Base de datos: Northwind
Pistas:
  • Crea #ListaPrecios con columnas ProductID INT, ProductName NVARCHAR(40), UnitPrice MONEY, CategoryID INT, SupplierID INT.
  • Inserta algunos productos existentes con precios diferentes y un producto ficticio nuevo (ej: ProductID = 999).
  • Compara por ProductID en la cláusula ON.
  • Usa WHEN MATCHED AND destino.UnitPrice <> origen.UnitPrice.
Ver solución
USE Northwind;
GO

-- Crear tabla temporal con la lista nueva de precios
CREATE TABLE #ListaPrecios (
    ProductID   INT PRIMARY KEY,
    ProductName NVARCHAR(40) NOT NULL,
    UnitPrice   MONEY NOT NULL,
    CategoryID  INT,
    SupplierID  INT
);

-- Algunos productos existentes con precios modificados + uno nuevo
INSERT INTO #ListaPrecios VALUES
    (1, 'Chai',            20.00, 1, 1), -- precio original: 18.00 → sube a 20
    (2, 'Chang',           19.00, 1, 1), -- precio original: 19.00 → sin cambio
    (11, 'Queso Cabrales',  25.00, 4, 5), -- precio original: 21.00 → sube a 25
    (17, 'Alice Mutton',    42.00, 6, 7), -- precio original: 39.00 → sube a 42
    (999,'Café Venezolano', 15.50, 1, 1); -- producto nuevo: no existe en Products

-- MERGE: actualizar precios e insertar nuevos, sin eliminar
MERGE INTO Products AS destino
USING #ListaPrecios AS origen
ON destino.ProductID = origen.ProductID

WHEN MATCHED AND destino.UnitPrice <> origen.UnitPrice THEN -- solo si el precio cambió
    UPDATE SET destino.UnitPrice = origen.UnitPrice

WHEN NOT MATCHED BY TARGET THEN -- producto nuevo
    INSERT (ProductName, UnitPrice, CategoryID, SupplierID, Discontinued)
    VALUES (origen.ProductName, origen.UnitPrice, origen.CategoryID, origen.SupplierID, 0)

OUTPUT
    $action AS accion,
    COALESCE(inserted.ProductID, deleted.ProductID) AS ProductID,
    inserted.ProductName,
    deleted.UnitPrice AS precio_antes,
    inserted.UnitPrice AS precio_despues;
GO

-- Limpiar: eliminar la tabla temporal
DROP TABLE #ListaPrecios;

-- Verificar los cambios
SELECT ProductID, ProductName, UnitPrice
FROM Products
WHERE ProductID IN (1, 2, 11, 17) OR ProductName = 'Café Venezolano';

    (6, 'Meat/Poultry',  'Prepared meats'), -- sin cambio
    (7, 'Produce',      'Dried fruit and bean curd'), -- sin cambio
-- ¡Categoría 8 (Seafood) NO está! → se debería eliminar si no tiene productos
    (9, 'Technology',      'Electronic devices and accessories'), -- nueva
    (10, 'Personal Care',  'Health and beauty products');        -- nueva

MERGE INTO Categories AS destino
USING #CategoriasNuevas AS origen
ON destino.CategoryID = origen.CategoryID

WHEN MATCHED AND
    CAST(destino.Description AS NVARCHAR(MAX)) <> CAST(origen.Description AS NVARCHAR(MAX)) THEN
    UPDATE SET destino.Description = origen.Description

WHEN NOT MATCHED BY TARGET THEN
    INSERT (CategoryName, Description)
    VALUES (origen.CategoryName, origen.Description)

-- Solo elimina si la categoría no tiene productos asociados
WHEN NOT MATCHED BY SOURCE AND
    NOT EXISTS (SELECT 1 FROM Products WHERE CategoryID = destino.CategoryID) THEN
    DELETE

OUTPUT
    $action AS accion,
    COALESCE(inserted.CategoryID, deleted.CategoryID) AS CategoryID,
    COALESCE(inserted.CategoryName, deleted.CategoryName) AS CategoryName;
GO

-- Seafood (id 8) tiene productos → NO se eliminó (verificar)
SELECT * FROM Categories ORDER BY CategoryID;

DROP TABLE #CategoriasNuevas;

Ejercicio 5 — Sincronizar directorio de clientes (Northwind)

Enunciado:

La empresa recibe un archivo CSV con el directorio actualizado de clientes. Simula ese archivo con una tabla temporal #ClientesActualizados y usa MERGE sobre Customers para:

  • Actualizar ContactName, City y Country si alguno cambió.
  • Insertar clientes completamente nuevos.
  • No eliminar clientes existentes.
  • Generar un reporte con OUTPUT que incluya la acción, el CustomerID y los valores anteriores/nuevos del ContactName.
Base de datos: Northwind
Pistas:
  • CustomerID en Northwind es NCHAR(5), no un entero. Ej: 'ALFKI'.
  • Usa ON destino.CustomerID = origen.CustomerID.
  • Para el WHEN MATCHED AND, compara los tres campos con OR.
  • Para el nuevo cliente, inventa un CustomerID de 5 caracteres (ej: 'VENZL').
  • El campo CompanyName es NOT NULL — no olvides incluirlo en el INSERT.
Ver solución
USE Northwind;
GO

-- Simular el archivo CSV con una tabla temporal
CREATE TABLE #ClientesActualizados (
    CustomerID  NCHAR(5) PRIMARY KEY,
    CompanyName NVARCHAR(40) NOT NULL,
    ContactName NVARCHAR(30),
    City        NVARCHAR(15),
    Country     NVARCHAR(15)
);

INSERT INTO #ClientesActualizados VALUES
    ('ALFKI', 'Alfreds Futterkiste',   'Maria Hernández', 'Munich',     'Germany'), -- ContactName y City cambian
    ('ANATR', 'Ana Trujillo Emparedados', 'Ana Trujillo',    'México D.F.', 'Mexico'), -- sin cambio
    ('BLONP', 'Blondel père et fils',   'Frédérique Citeaux', 'Paris', 'France'), -- City cambia (Strasbourg→Paris)
    ('VENZL', 'Venezolana de Alimentos', 'José Martínez',   'Caracas',   'Venezuela'); -- cliente nuevo

MERGE INTO Customers AS destino
USING #ClientesActualizados AS origen
ON destino.CustomerID = origen.CustomerID

-- Actualizar solo si algún campo cambió
WHEN MATCHED AND (
    ISNULL(destino.ContactName, '') <> ISNULL(origen.ContactName, '') OR
    ISNULL(destino.City, '') <> ISNULL(origen.City, '') OR
    ISNULL(destino.Country, '') <> ISNULL(origen.Country, '')
) THEN
    UPDATE SET
        destino.ContactName = origen.ContactName,
        destino.City        = origen.City,
        destino.Country     = origen.Country

-- Cliente nuevo → insertar
WHEN NOT MATCHED BY TARGET THEN
    INSERT (CustomerID, CompanyName, ContactName, City, Country)
    VALUES (origen.CustomerID, origen.CompanyName, origen.ContactName, origen.City, origen.Country)

-- Sin WHEN NOT MATCHED BY SOURCE → no se elimina ningún cliente

OUTPUT
    $action AS accion,
    COALESCE(inserted.CustomerID, deleted.CustomerID) AS CustomerID,
    deleted.ContactName AS contacto_antes,
    inserted.ContactName AS contacto_despues,
    deleted.City AS ciudad_antes,
    inserted.City AS ciudad_despues;
GO

-- Verificar resultados
SELECT CustomerID, CompanyName, ContactName, City, Country
FROM Customers
WHERE CustomerID IN ('ALFKI', 'ANATR', 'BLONP', 'VENZL');

DROP TABLE #ClientesActualizados;
